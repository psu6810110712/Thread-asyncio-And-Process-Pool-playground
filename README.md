## 📖 ทำความเข้าใจ Python Concurrency & Parallelism

ในโลกของการเขียนโปรแกรม การทำ **Concurrency** (การจัดการหลายงานพร้อมกัน) และ **Parallelism** (การประมวลผลหลายงานในเวลาเดียวกันจริง ๆ) เป็นสิ่งสำคัญมากในการเพิ่มประสิทธิภาพชองโปรแกรม โดยเฉพาะใน Python ซึ่งมีข้อจำกัดเรื่อง **GIL (Global Interpreter Lock)** ที่ทำให้ Thread หลายตัวไม่สามารถรันคำสั่ง Python พร้อมกันเป๊ะ ๆ ได้ใน 1 Process

โปรเจกต์นี้จึงจำลองสถานการณ์เพื่อเปรียบเทียบ 3 แนวทางหลักใน Python:

### 1. Asyncio (Asynchronous I/O)
- **เหมาะสำหรับ** งานประเภท **I/O Bound** เช่น การโหลดหน้าเว็บ, ดึงข้อมูลจาก Network หรือ API request
- **หลักการทำงาน** ทำงานอยู่บน **Single-thread** (Thread เดียว) แต่ใช้กลไกที่เรียกว่า **Event Loop** เมื่อฟังก์ชันหนึ่งต้อง "รอ" (หน่วงเวลา) มันจะใช้คำสั่ง `await` เพื่อพักงานนั้นไว้ชั่วคราว และสลับไปทำงานฟังก์ชันอื่นที่พร้อมรันต่อทันที
- **ข้อดี** ประหยัด Memory มาก ไม่ต้องสร้าง Thread หรือ Process ใหม่ให้หนักเครื่อง และรอดพ้นจากปัญหา Race Condition (เพราะมี Thread เดียว)
- **ในโค้ดจำลองด้วย** `asyncio.sleep()` เพื่อจำลองการรอผลจาก Network

### 2. Threading
- **เหมาะสำหรับ** งานประเภท **I/O Bound** เช่น การอ่าน/เขียนไฟล์ลง Disk (File I/O) หรือ Query Database
- **หลักการทำงาน** สร้าง OS Threads ย่อย ๆ ภายใน Process เดียวกัน แม้ว่า Python จะมี GIL ที่ล็อกให้รันโค้ดได้ทีละ Thread แต่เมื่อ Thread ไหนติดสถานะ I/O (เช่น รอเขียนไฟล์เสร็จ) GIL จะถูกปล่อยให้ Thread อื่นเข้ามาทำงานต่อได้ทันที
- **ข้อดี** ทำงาน File I/O ได้รวดเร็วขึ้นมากเมื่อเทียบกับการทำทีละฟังก์ชัน (Synchronous) อย่างรอคอยคิว
- **ข้อควรระวัง** เสี่ยงต่อปัญหา **Race Condition** หากหลาย Thread พยายามเข้าถึงและแก้ไขตัวแปรเดียวกันพร้อมกัน (อาจต้องใช้ `Lock` ช่วยจัดคิว)
- **ในโค้ดจำลองด้วย** `time.sleep()` และเขียนผลลัพธ์ลง Array ร่วมกัน

### 3. Multiprocessing (Process Pool)
- **เหมาะสำหรับ** งานประเภท **CPU Bound** เช่น การคำนวณทางคณิตศาสตร์หนัก ๆ, การแปลงไฟล์วิดีโอ (Video Processing), หรือ Machine Learning
- **หลักการทำงาน** สร้าง **Process ใหม่** ในระดับ OS ขึ้นมาเลย แต่ละ Process จะมี Python Interpreter และ Memory ของตัวเองแยกขาดจากกัน (ข้ามข้อจำกัดของ GIL ไปได้เลย) ทำให้สามารถประมวลผลขนานกันได้เต็มที่บน CPU หลาย Core (Parallelism แท้จริง)
- **ข้อดี** ใช้งานพลังของ CPU ได้ครบทุก Core ประสิทธิภาพสูงสำหรับการคำนวณที่ซับซ้อน
- **ข้อควรระวัง** การปั๊ม Process ใหม่มีค่าใช้จ่ายสูง (Overhead) ทั้งเวลาเริ่มต้น, ปริมาณ Memory ที่กินเบิ้ลขึ้นไป และการส่งข้อมูลข้าม Process (IPC - Inter-Process Communication) ทำได้ยากและช้ากว่า Thread
- **ในโค้ดจำลองด้วย** วงลูป `for` บวกลบคูณหารตัวเลขจำนวนมาก ๆ เพื่อให้ CPU ทำงานหนักจริง ๆ ค้างเอาไว้

## 📊 การวิเคราะห์ผลลัพธ์ (Performance Analysis)

เมื่อรันสคริปต์ `media_processor.py` คุณจะสังเกตเห็นผลลัพธ์ประมาณนี้:

```text
⏱️  Asyncio (I/O): 0.4858 seconds
⏱️  Threading (I/O): 0.2056 seconds
⏱️  Process Pool (CPU): 0.8846 seconds
```

**ทำไมผลลัพธ์ถึงเป็นแบบนี้?**
1. **Threading เร็วที่สุดในเคสนี้** เพราะงานส่วนใหญ่ของเราจำลองด้วยการหน่วงเวลา (`time.sleep()`) Thread สามารถเกิดและรันสลับกันได้ไวมากด้วย Overhead ในระดับ OS ที่ต่ำ
2. **Asyncio ตามมาติดๆ** แม้จะเป็น Thread เดียว แต่ไม่ต้องเสียเวลาให้ OS มาช่วยสลับ Thread หรือ Process อย่างไรก็ตาม โค้ดที่เรารันมี Overhead เล็กน้อยจากการจัดการคิวของ Event Loop ขาประจำและ Coroutine Tasks
3. **Multiprocessing ช้าที่สุด** นี่คือจุดสำคัญ งานที่เราจำลอง (บวกเลขเล่น) ไม่ได้หนักพอที่จะคุ้มค่ากับการเสียเวลา **Overhead ในการสร้าง Process ใหม่** หากเราแก้ไขโค้ดโดยเพิ่มรอบการบวกเลขเป็น `10**8` แทน Multiprocessing จะกลับมาทำเวลาได้ดีที่สุดและทำได้ดีกว่าวิธีอื่นๆ
